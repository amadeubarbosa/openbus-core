\section{O que é o Openbus ?}
O Openbus é uma arquitetura aberta e orientada a serviços 
(\emph{Service-Oriented Architecture}, ou SOA) para integrar dados e aplicações
heterogêneas de domínios distintos ou não. A arquitetura é baseada em um 
\emph{barramento} em que os usuários se conectam para consumir ou prover 
serviços.

\section{Por que utilizar o Openbus ?}
Quando há necessidade de integração entre aplicações, algumas soluções podem 
ser apresentadas conforme a complexidade do ambiente a ser integrado. 
Caso o problema se resuma a troca de dados entre as aplicações, uma primeira 
solução é a integração externa, que consiste na exportação do dado em um 
determinado formato padronizado que por sua vez possa ser carregado pela
aplicação de destino. Este processo é normalmente manual, deixando a cargo 
do usuário ou, do \emph{job} que automatiza o processo, a responsabilidade pela
sua execução. Um outro problema é a possibilidade de ocorrer perda de 
informação quando o formato padrão utilizado não consegue representar 
corretamente o dado de origem. 

As opções de soluções disponíveis para a integração, podem ser ampliadas 
quando há a presença de aplicações de código aberto ou que ofereçam
interfaces de programação. Para estes casos, uma integração programática 
pode ser realizada de forma direta, através de uma implementação que  
permita ao usuário comandar a transferência de dados. Este modo de interação 
agrega mais qualidade, diminuindo as chances de perda de informação durante a 
transferência do dado, pois os metadados das aplicações envolvidas podem ser 
considerados. Além da troca de dados, há a possibilidade 
de troca de mensagens entre as aplicações. O problema desta solução é que 
uma ponte deve ser desenvolvida para cada par de aplicações, o que resulta em 
um número quadrático de codificações de pontes em relação ao número de 
aplicações. Por exemplo, para integrar 10 aplicações precisamos implementar 
45 pontes. Além disso, a troca de mensagens é naturalmente 
limitada entre cada par de aplicações.

Os problemas apresentados acima pela programática direta, podem ser eliminados 
com a adoção de uma plataforma comum de integração entre aplicações. Como por 
exemplo um \emph{barramento}, em que cada aplicação se conecta 
a este, segundo um protocolo próprio deste barramento, e carrega ou 
exporta dados para outras aplicações também conectadas ao barramento. Essa
arquitetura define uma linguagem comum entre todas as aplicações, e, induz o 
compartilhamento de interfaces bem definidas para os serviços comuns entre as 
aplicações envolvidas, desta forma o esforço de codificação das pontes 
pode ser reduzido para um esforço linear, ou seja, para se integrar 
\emph{n} aplicações, \emph{n} pontes são necessárias.
No mais, a troca de mensagens dinâmicas entre todas as aplicações passa 
a ser possível, possibilitando um ambiente de colaboração entre as aplicações.

\section{Por que eu preciso do Openbus se eu tenho CORBA?}

É verdade que a arquitetura de CORBA por si só já permite que o cliente possa
integrar aplicações desenvolvidas em diferentes linguagens, e, que estejam 
operando em um ambiente multi-plataforma.
A questão é que para um ambiente composto por um número significativo de 
aplicações, a presença de um elemento centralizador como um barramento, pode 
ofertar uma estrutura que permita um aumento da qualidade e uma redução de 
esforço na integração.
Uma integração segura e eficiente de um ambiente complexo certamente traz à 
tona algumas preocupações, como por exemplo: o controle de acesso aos serviços
ofertados por cada usuário, a gestão dos serviços oferecidos, a facilidade 
de se agrupar serviços de mesma natureza e a troca de mensagens entre todas 
as aplicações envolvidas. 

O Openbus já oferece um grupo básico de serviços 
para que o usuário não tenha esforço na elaboração e implementação de serviços
destinados ao auxílio da integração, e, que são necessários para uma 
integração de qualidade satisfatória. Quando o barramento não oferece um 
serviço específico de suporte a integração, a arquitetura permite que o cliente
desenvolva ou anexe um serviço externo que desempenhe a função desejada.


Sendo assim o Openbus pode ser posicionado como um \emph{middleware} que está 
em um nível de abstração maior do que CORBA, oferendo aos seus usuários uma 
solução de integração menos custosa e escalável por utilizar um sistema 
de componentes(SCS) e serviços básicos de auxílio.

\section{Por que CORBA e não \emph{Web services} ?}

CORBA é uma tecnologia mais madura e completa do que \emph{Web services},
apresentando características que possibilitam uma integração envolvendo 
escalabilidade e desempenho.
Diversos pontos podem ser levantados para justificar a motivação de se 
escolher CORBA frente \emph{Web services}. Três importantes aspectos que 
foram avaliados na tecnologia \emph{Web services}:
\begin{enumerate}
  \item Não há noção de objeto, referência a objetos ou clico-de-vida, ou seja,
  não temos características fundamentais de um sistema de objetos distribuídos.
  Desta forma, por exemplo, não há a estrutura necessária para se manter um 
  estado persistente durante as interações remotas.
  \item A linguagem de descrição IDL é muito mais natural para o mapeamento 
  de uma interface nativa escrita em uma linguagem orientada a objeto, do que 
  a linguagem WSDL utilizada pelo \emph{Web services}.
  \item Presença de \emph{overhead} não satisfatório no processo de 
  \emph{marshaling}, em que é necessária a elaboração de um documento XML 
  para representar a informação a ser transmitida. No mais, uma \emph{stream} 
  de dados binários não pode ser passada de forma direta. 
\end{enumerate}
